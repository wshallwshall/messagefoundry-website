Security & PHI

Engineered for PHI — and built to recognized standards

For a healthcare integration engine, security isn't a feature — it's the foundation. MessageFoundry is built to a published Secure Development Standard and measured against the frameworks a CISO already trusts: NIST SSDF, NIST 800-53 / 800-66 (the HIPAA Security Rule), and OWASP ASVS 5.0 Level 2. It's secure by default, encrypted end to end, and clear about who's responsible for what.

Built to recognized standards

Mapped to the frameworks your auditors use

Four bodies of work answer four questions — how it's built, how it's tested, what controls it implements, and how it's independently verified.

NIST SSDF — built securely

The secure software development lifecycle follows NIST SP 800-218 (SSDF) across all four practice groups — prepare, protect, produce, and respond — with threat modeling per interface, peer review, and secure-by-default design.

NIST 800-115 — tested rigorously

Security testing follows NIST SP 800-115: automated SAST / SCA / secret-scanning on every build, dynamic testing per release, and independent source review + penetration testing before production.

NIST 800-53 / 800-66 — HIPAA-mapped

Implemented controls are drawn from the NIST SP 800-53 catalog and mapped to the HIPAA Security Rule via NIST SP 800-66 Rev. 2 — access control, audit, integrity, and transmission security.

OWASP ASVS 5.0 Level 2 — verified

The application is verified against OWASP ASVS 5.0 Level 2 — the standard level for software handling sensitive data — scoped chapter-by-chapter, with any exclusions documented and justified.

Honest by design. NIST doesn't issue certificates for these frameworks, so we don't claim one. These are alignment claims backed by evidence — the Standards document, test reports, the ASVS attestation, and an SBOM, published with our releases. "Built to," "verified against," "mapped to" — never "certified."

Secure by default

The safe configuration is the default one

You don't have to harden it — it ships hardened. TLS is on, encryption is on, accounts are least-privilege, and audit logging is verbose out of the box. Insecure options require an explicit, documented opt-in.

  • TLS everywhere — all network communication over TLS 1.2+ (prefer 1.3), including MLLP-over-TLS and mTLS for system-to-system links. No cleartext PHI on the wire.
  • Encryption at rest, on by default — sensitive data encrypted with approved algorithms (SQLCipher / database-native), with FIPS-validated crypto where a deployment requires it.
  • Deny-by-default RBAC — role-based, least-privilege access enforced on every action, with per-channel data-scope confinement.
  • Tamper-resistant audit — a timestamped, user-attributed audit log; no PHI or secrets in logs, ever.
  • Fail-closed — errors fail safe; faults never leak stack traces or sensitive data.
  • Secrets stay out of code — sourced from an environment / secret store, enforced by pre-commit and CI secret scanning; the git history is kept clean.
Interface authentication

Modern auth for every connection — not a shared password

An integration engine authenticates systems, not people. Each connection uses the strongest mechanism its partner supports, recorded in the connection definition.

Preferred — system-to-system

  • Mutual TLS (mTLS) — client-certificate auth with full chain + revocation checks; sender-constrained tokens preferred.
  • OAuth 2.0 client-credentials — asymmetric (private_key_jwt), short-lived, per-connection scoped tokens.
  • SMART on FHIR — Backend Services profile (signed JWT assertion, system/ scopes) for any FHIR interface.

Active Directory / enterprise

  • gMSA service account — auto-rotated password, never stored in config.
  • Kerberos / Integrated Windows Auth (NTLM disabled where feasible); integrated DB auth.
  • LDAPS only for directory lookups; roles mapped to AD groups; human operators federate via AD FS / Entra ID (OIDC / SAML).

Legacy — documented per connection

  • HTTP Basic over TLS, per-connection API keys, or SOAP WS-Security (X.509-signed preferred).
  • Used only when a partner can't support a preferred mechanism — always over TLS, vaulted, scoped, rotated, and recorded.

Across every mechanism: TLS everywhere, credentials and keys in a secret store (never in code or config), per-connection least privilege, and per-connection IP allow-listing / network segmentation as defense-in-depth.

Shared responsibility

Clear about who owns what

The software is built by one party and deployed by another. Stating the split plainly is itself a security control — neither side assumes the other has it covered.

MessageFoundry (the project)

  • Secure development practices and secure-by-default configuration.
  • Security testing and attestation of the software.
  • Vulnerability response and coordinated disclosure.
  • Documentation, evidence, and published attestations.

Your organization (the deployer)

  • Your environment, host, and network security.
  • Identity, credential, and key management in your environment.
  • Backups, disaster recovery, and availability.
  • Your HIPAA program — risk analysis, Business Associate Agreements, and the deployment's compliance.

Said plainly. The software supports a HIPAA-compliant deployment — it doesn't make you compliant on its own. Our attestation that the software was built securely is evidence for your assessment, not a substitute for it.

Secure SDLC & supply chain

Provenance you can verify

Open-source software is only as trustworthy as the pipeline that builds it. Ours is built to be auditable end to end.

  • Signed, verifiable releases — each release is versioned, signed, and accompanied by a software bill of materials (SBOM), so adopters can verify provenance and integrity.
  • CI security gates — SAST, software-composition analysis, and secret scanning run on every change; builds fail on new high / critical findings.
  • Reviewed, protected code — branch protection, required peer review, signed commits, no direct commits to main.
  • Coordinated disclosure — a published SECURITY.md with a private reporting channel and a disclosure timeline feeds a tracked remediation process.
  • No real PHI in development — synthetic or de-identified data only, on disk-encrypted developer machines.
