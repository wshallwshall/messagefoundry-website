Features

Everything you need for reliable point-to-point HL7

A small, sharp feature set focused on doing the common interfaces well — durable delivery, tolerant parsing, code-first routing, and PHI-aware operations. Items marked planned are on the roadmap.

Reliability

Durable by construction

The message store is the queue. A transactional inbox/outbox on SQLite (WAL) is the reliability backbone — no separate broker to run, patch, or monitor.

  • At-least-once delivery — the inbound is ACKed only after the message and its outbox rows are committed.
  • Per-outbound workers — each destination drains independently; one slow or failing peer never blocks the others.
  • Retries with backoff and automatic dead-lettering for messages that can't be delivered.
  • Replay — re-drive a stored message through delivery from the console or API.
  • Asyncio core — one listener per inbound, one worker per outbound, supervised so a crash in one is isolated.
Parsing & validation

Two-tier, by design

Fast and tolerant on the hot path; strict and version-aware only where you ask for it. Real-world HL7 is frequently non-conformant — the engine is built to expect that.

  • python-hl7 for fast field peek during routing and filtering.
  • hl7apy for opt-in, version-aware strict validation, per inbound connection — kept off the routing path.
  • Encoding from MSH — field, component, repetition, escape, and subcomponent separators are read from the message, never hardcoded.
  • Raw message preserved alongside the transformed form, so an operator always sees exactly what arrived.
  • Parse failures are routed to the error / dead-letter path and logged as ERROR — a bad message never crashes a connection.
Connections

Pluggable transports, resolved through a registry

New connection types are registered, not special-cased in the pipeline.

MLLP built

Correct 0x0B … 0x1C 0x0D framing, inbound and outbound. ACK/NAK generation (AA / AE / AR) with the ack mode — original, enhanced, or none — configurable per inbound connection.

File built

File source and destination, with templated output names (e.g. {MSH-10}.hl7) drawn from message fields.

TCP · HTTP · DB planned

Additional transports — plus REST/FHIR destinations and a database poller — are on the roadmap, implemented against the same connector contract.

Count-and-log invariant. Every message a connection takes in or puts out is counted and logged — a router or filter decision is a recorded outcome (UNROUTED / FILTERED), never a silent drop.

Routing & handling

Code-first, pure functions

A Router decides forwarding (and optional filtering); a Handler filters, transforms, then returns Sends. They're pure Python functions, registered by name — no declarative filter/transform DSL to learn.

  • @router returns the handler name(s) a message should go to — or an empty list to leave it unrouted.
  • @handler returns None to filter, or one or more Sends addressed to outbound connections.
  • Wired by name — inbound names a Router, Routers name Handlers, Handlers send to outbound Connections.
  • Shared helpers live in _-prefixed modules the loader skips, so config stays DRY.
  • Transforms stay pure where possible — message in, message out; side effects belong in connections.
Security & PHI

PHI is a first-class concern

This engine carries PHI, so authentication, authorization, audit, and encryption-at-rest are built in — not bolted on. The API binds 127.0.0.1 by default and requires authentication.

  • Authentication — local users and Active Directory (LDAP / Kerberos), with opaque sessions.
  • RBAC — fixed built-in roles with deny-by-default, per-route permissions, including per-channel scoping.
  • Audit log — every PHI access (raw view, summary display, replay) is recorded with the acting user.
  • Encryption-at-rest — message bodies are encrypted with AES-256-GCM in the store.
  • Roadmap planned — log redaction and MLLPS / TLS for remote exposure.
Operate & build

A console and an editor, both talking to one API

PySide6 admin console

Connection dashboard, message browser, an HL7 parse-tree viewer, delivery and audit trail, and one-click replay. A separate process that reaches the engine only over the localhost API.

VS Code extension

Author and test interfaces in your editor: a New Route Wizard, validate-on-save, a Test Bench that dry-runs .hl7 files with before/after diffs, Stage → Promote to a running engine, and an HL7-aware chat participant.

CLI & localhost API

messagefoundry serve runs the engine and exposes connection control, message search/detail, replay, stats, and a live WebSocket feed. check gates commits; generate produces synthetic, PHI-free HL7.

Deployment

Runs headless, where your data lives

On-premises by default — no PHI leaves the local environment without explicit, reviewed configuration.

  • Windows service via NSSM — install/uninstall scripts included.
  • Headless engine — no GUI imports; testable and runnable as a service.
  • SQLite (WAL) today; additional store backends are in progress in progress.
  • One config, many environments — DEV/PROD values layer over the same Python modules.
