Optional AI assistance

AI is optional — and it never touches your messages

MessageFoundry needs no AI to run. The engine moves, routes, and transforms every message on its own, and the engine never sends message traffic to an AI — there is no model anywhere in the message path.

AI shows up only at the keyboard, as an optional coding aid. The MessageFoundry VS Code extension adds a chat assistant that works through whatever model you've already set up in your editor — Copilot, Claude, and others — so MessageFoundry bundles no AI vendor and ships no keys. It helps write and edit the Python that drives each feed, with a developer reviewing every line.

We build MessageFoundry's own feeds this way too. Prefer to skip AI entirely? Guided wizards and plain Python do the same job.

# @messagefoundry · /transform
# "Map the ADT patient name to a FHIR Patient."

@handler("to_fhir")
def to_fhir(msg):
    patient = {
        "resourceType": "Patient",
        "name":       msg["PID-5"],
        "identifier": msg["PID-3"],
        "birthDate":  msg["PID-7"],
    }
    return Send("FHIR_Out", patient)

# What the assistant is given:
#   - your code + connection / route names
#   - never a message body, never PHI
# You review every line. Runs on your model.

In your editor

A chat assistant that speaks MessageFoundry

Install the MessageFoundry VS Code extension and you get a @messagefoundry chat participant. Ask it about an interface in plain language — it's primed with MessageFoundry's real Router and Handler API, so its answers and drafts use the actual msg[...] and Send(...) surface, not generic, almost-right guesses.

It runs on the model you already use in VS Code's Chat — Copilot, Claude, whatever you've configured — so there's no vendor to adopt and no API key for MessageFoundry to hold. Each request goes straight from your editor to that provider; the engine is never in the loop.

Three commands, primed with the API

  • /explain — walk through what a router or handler does, in plain language.
  • /transform — draft a @handler transform from a description of the mapping you need.
  • /review — get a review of the handler you're writing, against the real MessageFoundry API.

Provider-agnostic by design — you bring the model; MessageFoundry holds no keys.

Why it's fast

Your logic becomes plain, reviewable Python

Legacy engines lock integration logic inside proprietary scripts and GUIs — JavaScript channels, Tcl, ObjectScript, XML buried in a database. Porting it by hand means relearning each system and rebuilding interface by interface.

MessageFoundry's configuration is code, so the assistant can do the mechanical translation for you: paste a legacy channel or open its export, and it drafts the matching inbound / outbound Connections, a @router, and the @handler transforms in Python. You stay in control — every line lands as a diff you read, adjust, and approve.

What the assistant drafts for you

Point it at one legacy channel and it writes a first-pass version of each piece in Python — yours to read, adjust, and approve before anything runs:

  • Connections — MLLP, TCP, X12/EDI, HTTP/REST, SOAP, FHIR REST, database, and file endpoints (local and SFTP/FTP/FTPS) mapped from your existing source/destination config.
  • Routing — a @router that mirrors your channel's filter and routing rules.
  • Transforms — @handler functions that reproduce field mappings and reshaping as pure Python.
  • Tests — starting points for harness scenarios so you can prove parity before cutover.
Legacy → modern

The same interface, two eras of tooling

The legacy way

  • Logic locked in a proprietary language (JavaScript, Tcl, ObjectScript) and a vendor GUI.
  • Configuration exported as XML or held in a database — hard to diff or review.
  • Migration is manual re-platforming, channel by channel, by scarce specialists.
  • Testing and promotion live in separate, proprietary tooling.

The MessageFoundry way

  • Logic is plain Python in your git repository — read it, diff it, own it.
  • The AI assistant drafts that Python from your legacy logic, in your editor.
  • Review as a pull request; dry-run in the included test harness on synthetic data.
  • Stage → Promote to a running engine — no separate platform to learn.
The migration loop

From legacy channel to running route

Point the assistant at your legacy logic

Open or paste an existing channel — a Mirth transformer, a Cloverleaf Tcl proc, a Rhapsody filter. That's source code, so it stays within the assistant's code-only scope.

It drafts the MessageFoundry equivalent

You get Python Connections, a Router, and Handlers that mirror the original's routing, filtering, and field mapping — wired by name, ready to read.

Review it like any code change

The draft lands as a diff in your editor and your pull request. Adjust, comment, and approve — the migration is now a normal review, not a black box.

Prove parity in the test harness

Dry-run synthetic, PHI-free messages through the new route and compare before/after. The headless scenario runner keeps the check green in CI.

Stage → Promote, then repeat in parallel

Promote the verified route to a running engine. Because each interface is independent code, a team can migrate many channels at once instead of serially.

Governed & PHI-safe

One central policy — never your messages

The assistant carries PHI implications, so whoever operates the install sets the policy — not each developer. A central “off”, or a cap on what the assistant may ever see, is honored on every workstation, and the editor resolves that policy before every request.

Two dials, one ceiling

The policy is two independent settings:

  • mode — whether and how AI runs: off, byo (bring-your-own — the path that works today), or an engine-brokered managed mode, including a BAA-backed variant (both on the roadmap).
  • data_scope — the most sensitive data the assistant may ever see, lowest to highest: code_only → synthetic → de-identified → phi.
  • A production posture sets a ceiling on data_scope — never on mode. Production can cap the data, and a central off can’t be overridden anywhere.

The effective policy is computed by clamping your request down to what’s allowed — and every adjustment is recorded with a plain-English reason you can read.

How it’s enforced

  • Code-only, today — in practice the assistant attaches only code_only context: the names in your config graph (connections, routers, handlers) plus a capped slice of the file you’re editing. No message bodies or patient data are ever sent, whatever the mode or provider.
  • RBAC-gated — an ai:assist permission controls who may use it, granted to the Coding and Administrator roles only — deny-by-default, like the rest of the engine.
  • Fails safe — the editor reads the policy from a no-auth GET /ai/policy (so even a token-less client still obeys a central “off”), with an offline messagefoundry ai-policy CLI fallback and a conservative default. It disables itself if AI is off, if it can’t service a managed mode, or if you lack the permission — and never silently downgrades a managed mode to bring-your-own.

On the roadmap: an engine-brokered managed mode, including a BAA-backed variant that would be the only way the assistant could ever be given PHI. The de-identified scope waits on that work and falls back automatically until then — so today, the assistant stays strictly code-only.

Migrating doesn't expose patients. Channel logic is code, and code is all the assistant sends. Your historical messages and live PHI never go to a model during a migration — the work happens entirely on the integration logic.
